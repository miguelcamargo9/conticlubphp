# ContiClub API

Laravel 6.x backend for the ContiClub tire loyalty program. Users submit invoices for tire
purchases, earn points based on their profile + tire type, and redeem those points for
products from the catalog. This repo contains **only the API**; the web/mobile clients live
in separate repositories.

This README is the internal onboarding doc for developers working on this service. For the
day-to-day assistant guide used by Claude Code, see [`CLAUDE.md`](./CLAUDE.md) ‚Äî the two
documents intentionally overlap so each stands on its own.

## Stack

| Layer     | Version                                                                                         |
|-----------|-------------------------------------------------------------------------------------------------|
| Language  | **PHP 7.2.34** (production runtime)                                                             |
| Framework | Laravel 6.x                                                                                     |
| Auth      | Laravel Passport 9.x (OAuth2 password grant)                                                    |
| Database  | MySQL 8.0.45 (AWS RDS in prod, `mysql:8.0.45` container locally)                                |
| Storage   | AWS S3 via `league/flysystem-aws-s3-v3` for invoice images                                      |
| HTTP      | Nginx + PHP-FPM (see `docker-compose.yml` / `Dockerfile` / `nginx_config.conf`)                 |

### Why we're stuck on PHP 7.2 / Laravel 6

Laravel 6 reached EOL in June 2022 and PHP 7.2 is long out of support, but the production
runtime on AWS is still PHP 7.2.34. To keep local and CI installs resolving against the
same dependency tree, `composer.json` pins:

```json
"require": { "php": "^7.2.5" },
"config": { "platform": { "php": "7.2.34" } }
```

**Do not** bump `require.php` or remove the platform override without first coordinating a
production PHP upgrade. Several dependencies are also held at their last L6-compatible
versions and are listed under `audit.ignore` in `composer.json`. The next upgrade cycle
should target Laravel 11/12 per the original migration proposal (Option B).

## Quickstart (Docker)

The docker-compose stack is the recommended local setup ‚Äî it matches the MySQL version
used in production and avoids fighting with a newer local PHP.

```bash
# 1. Clone and enter the project
git clone <repo-url> conticlubphp && cd conticlubphp

# 2. Install PHP deps (runs inside a PHP 7.2 container via the image ‚Äî see below for hostside option)
composer install

# 3. JS deps (only needed if you touch the Blade-based landing pages / mix pipeline)
npm install

# 4. Bootstrap the environment file
cp .env.example .env
php artisan key:generate

# 5. Bring up db + php-fpm + nginx
docker-compose up -d

# 6. Run migrations + seed reference data
docker-compose exec php php artisan migrate --seed

# 7. Install Passport keys (first time only)
docker-compose exec php php artisan passport:install
```

The API is now reachable at **http://localhost:8080** and MySQL at **localhost:3307**
(user `laravel_user`, password `notsecret`, db `conticlub` ‚Äî see `docker-compose.yml`).

### Host-side (no Docker)

If you run PHP/MySQL directly on the host, you need PHP 7.2.x available. Everything else
mirrors the Docker flow ‚Äî just skip `docker-compose` and run the artisan commands
directly:

```bash
composer install
cp .env.example .env && php artisan key:generate
php artisan migrate --seed
php artisan passport:install
php artisan serve   # http://localhost:8000
```

### Required environment variables

| Var                      | Purpose                                                              |
|--------------------------|----------------------------------------------------------------------|
| `DB_HOST` / `DB_PORT` / `DB_DATABASE` / `DB_USERNAME` / `DB_PASSWORD` | MySQL connection. Docker uses host `db`, port `3306` internally. |
| `CLIENT_KEY`             | Shared secret enforced by the custom `client` middleware on public registration endpoints. |
| `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` / `AWS_DEFAULT_REGION` / `AWS_BUCKET` | S3 credentials for invoice image uploads. |
| Laravel Passport keys    | Generated by `php artisan passport:install` into `storage/`.         |

## Testing

PHPUnit is configured with two suites (`Unit` and `Feature`), but only the `Feature` suite
has tests today. Feature tests run against the real dev database with `DatabaseTransactions`
‚Äî there is **no** sqlite-in-memory config, so `.env.testing` must point at a usable MySQL.

```bash
# All tests
./vendor/bin/phpunit

# Feature suite only (Unit suite is currently empty)
./vendor/bin/phpunit --testsuite=Feature

# Single file
./vendor/bin/phpunit tests/Feature/InvoicesTest.php

# Single method
./vendor/bin/phpunit --filter=test_method_name tests/Feature/InvoicesTest.php
```

Current coverage (`tests/Feature/`):

- `AuthTest` ‚Äî `/api/login` happy path and bad-credentials, Passport token issuance
- `UsersTest`, `SlidesTest`, `ProductsTest` ‚Äî CRUD read endpoints behind `auth:api`
- `InvoicesTest` ‚Äî full invoice create/approve/reject flow, including points deduction
  and auto-rejection of pending `ChangePoints` requests when a user's balance drops

When adding tests, follow the established pattern: `use DatabaseTransactions`,
authenticate with `Passport::actingAs($user)`, and rely on **seeded reference data**
(tires, profiles, `TirePointsByProfile`) ‚Äî the project does not ship model factories for
domain tables.

## Architecture

### Authentication

- **Laravel Passport** (OAuth2 password grant). `POST /api/login` issues access tokens via
  `AccessTokenController::issueToken`.
- **`auth:api` middleware** protects most endpoints.
- **`client` middleware** (custom) validates a shared `CLIENT_KEY` for public registration
  endpoints so the mobile app can create users without an access token.
- **`cors` middleware** handles cross-origin preflight.
- **User profiles** (`users.profiles_id`) act as coarse roles ‚Äî e.g. profile `4` is the
  admin role allowed to approve/reject invoices. Profiles also drive how many points each
  tire is worth (see `TirePointsByProfile`).

### Core domain model

```
Brand ‚îÄ‚îÄ hasMany ‚îÄ‚îÄ‚ñ∂ Design ‚îÄ‚îÄ hasMany ‚îÄ‚îÄ‚ñ∂ Tire
                                             ‚îÇ
                                             ‚îÇ hasMany
                                             ‚ñº
                                     TirePointsByProfile  ‚óÄ‚îÄ‚îÄ profile determines point value
```

- **Brand ‚Üí Design ‚Üí Tire** is the tire classification hierarchy. `TirePointsByProfile`
  joins a tire, a user profile and the `total_points` awarded per unit.
- **Invoice** is submitted by a user with one or more `InvoiceReference` rows (tire +
  quantity + calculated points). State flows **pending ‚Üí `Aprobada` | `Rechazada`**. Sale
  date must be **‚â• 2023-05-01**. Images land in S3 at
  `/files/invoices/{user_id}/{invoice_id}-{filename}`.
- **Point** tracks points per invoice (`complete` or `rechazado`). `PointsMovement` +
  `PointsMovementsDetail` keep the full audit trail of point transactions.
- **ChangePoints** handles product redemption requests with states
  **`espera` ‚Üí `approved` | `rechazado` | `bought`**.
- **Product** has categories, images and a point cost.

### Business logic hotspots

Most domain logic lives directly in controllers (no service layer, no FormRequests).

**Invoice creation** ‚Äî `InvoiceController::create`:

1. Validate sale date ‚â• 2023-05-01.
2. Upload invoice image to S3.
3. For each tire reference, look up `TirePointsByProfile` for the user's profile. Reject
   the whole invoice if any tire is inactive / missing for that profile.
4. Points per line = `quantity √ó TirePointsByProfile.total_points`.
5. Create the `Invoice` + `InvoiceReference` rows, credit the user's point balance, write
   `Point` and `PointsMovement*` history records.

**Invoice rejection** ‚Äî `InvoiceController::rejected` (wrapped in `DB::beginTransaction()`):

1. Deduct the previously-credited points from the user.
2. Mark the `Point` row as `rechazado`.
3. Walk every pending `ChangePoints` request belonging to that user. If the user's
   post-deduction balance can no longer cover a pending request, auto-reject it.
4. Append a `PointsMovement` / `PointsMovementsDetail` history entry describing the
   rejection.

If you touch points math in `InvoiceController::create`, always re-check the rejection
path ‚Äî the two must stay in sync or users can end up with dangling `ChangePoints` they
can't afford.

### Database conventions

- **Singular** table names (`invoice`, `tire`, `product`, not `invoices`/`tires`/`products`).
- **Foreign keys** use the `{model}_id` pattern (`users_id`, `subsidiary_id`, `profiles_id`).
- Eloquent models in `app/Models/` declare `$fillable` explicitly and carry PHPDoc
  property blocks for IDE support ‚Äî regenerate via `php artisan ide-helper:models` after
  schema changes.

### Code patterns worth knowing

- Controllers pull the authenticated user with `Request::user()`.
- JSON payloads are read with `Input::post("data")` then `json_decode()` ‚Äî **not** via
  `$request->json()` / FormRequest. New endpoints should stay consistent with this
  convention unless you're intentionally modernising a whole controller.
- Responses are returned as plain arrays: `["message" => "success", "data" => ...]` or
  `["message" => "error", ...]`. There is no central JSON resource / transformer layer.
- Multi-step writes (invoice rejection, point corrections) must be wrapped in
  `DB::beginTransaction()` / `DB::commit()` / `DB::rollBack()`.
- Validation is done inline inside controllers ‚Äî there are no FormRequest classes.

## Common tasks

### Add a new API endpoint

1. Register the route in `routes/api.php` with the correct middleware (`auth:api` for
   everything user-facing, `client` only for CLIENT_KEY-protected public endpoints).
2. Add or extend a controller in `app/Http/Controllers/API/` following the existing naming
   and response conventions.
3. Touch models in `app/Models/` only if you need new columns or relationships.
4. Add a Feature test under `tests/Feature/` using `Passport::actingAs` + `DatabaseTransactions`.

### Change points math

- The source of truth is `TirePointsByProfile.total_points` per (tire, profile) pair.
- Calculation lives in `InvoiceController::create` (roughly lines 66-88).
- Always re-read `InvoiceController::rejected` afterwards ‚Äî the rejection path has to
  symmetrically undo whatever `create` did, or users end up with orphaned
  `ChangePoints` requests.

### Add a tire or product

- Tires: walk the **Brand ‚Üí Design ‚Üí Tire** hierarchy top-down, then create a
  `TirePointsByProfile` row for **every** profile that should earn points on it. Forgetting
  a profile silently breaks invoice submission for that user type.
- Products: create the `Product` + category + images, then verify redemption via
  `ChangePointsController`.

### Database schema changes

```bash
php artisan make:migration add_something_to_invoice_table
# edit the migration
php artisan migrate
# update $fillable on the model
php artisan ide-helper:models
```

## Troubleshooting

- **`composer install` fails with a PHP version mismatch** ‚Äî your host PHP is newer than
  7.2. That's fine for running the tooling, but you still need the platform pin to resolve
  dependencies. Don't remove it. If truly blocked, run `composer install` inside the
  `php` docker service: `docker-compose exec php composer install`.
- **Invoice creation returns an error about points** ‚Äî the submitted tire isn't active
  in `TirePointsByProfile` for the user's profile. Seed it or use a tire that is.
- **`auth:api` rejects requests** ‚Äî Passport keys are missing. Run
  `php artisan passport:install` (or `passport:keys` if the install already ran once).
- **S3 uploads fail locally** ‚Äî either configure real AWS credentials or point
  `FILESYSTEM_DRIVER` at `local` in your `.env` for dev; production expects S3.
